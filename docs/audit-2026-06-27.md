## 摘要（TL;DR）

1. **数据底盘比预期干净**：孤儿关系 0、重复关系 0（关系表均有复合主键）、JSON 全合法、integrity_check ok、来源字段填充率高（herbs/cases 的 source_repo 100%）。
2. **但推理链在最关键处断裂**：`症状→证型` 通（89%），`证型→方剂` **仅 6.2%（19/305）致命断**，`方剂→药物` 26%，`医案→方剂` 17%。**当前无法支撑可靠的症状→方剂推理**。
3. **方剂实体本身有问题**：≥9 个方剂重名（桂枝汤×5、小柴胡汤/白虎汤×4…），分「真重复」（同源同内容，ETL bug）和「多来源版本」（nihaixia 版带六经，可合并补字段）。
4. **server.py 有 4 类 P0 安全漏洞**：SQL 注入（table/列名拼接，4 个端点）+ 目录穿越（/screenshots/）。本地服务若绑 0.0.0.0 可被读取全库/任意文件。
5. **ETL 不可重复重建**：`populate.py` 的 `build()` 是空壳，README 宣称的 `--rebuild` 失效；`clinical_cases` 用裸 INSERT，重跑会重复。
6. **建议路线**：P0 安全修复（不改 DB）→ 建可信基线（审计脚本+API 测试）→ 工程化加固（config+etl.py）→ 数据治理（去重+六经+方证+evidence/aliases 表）→ 结构化查询。

## 2. 已确认 Bug（安全 / 工程）

### 🔴 P0-SEC-1　server.py SQL 注入 —— `api_browse` 表名拼接
- **证据**：`server.py:123` `conn.execute(f"SELECT COUNT(*) FROM {table} WHERE {where}", params)`、`:124` `f"SELECT * FROM {table} WHERE {where} LIMIT ? OFFSET ?"`、路由处 `table = params.get('table',...)[0]`（query string 直取）
- **触发**：`GET /api/browse?table=herbs%20UNION%20SELECT%20sql%20FROM%20sqlite_master--`
- **影响**：读取任意表结构/数据
- **修复**：白名单 `ALLOWED_TABLES`

### 🔴 P0-SEC-2　server.py SQL 注入 —— `api_browse` 过滤列名拼接
- **证据**：`server.py:120` `where += f" AND {k} LIKE ?"`，`k` 是任意非 table/page 的 query 参数名
- **触发**：`GET /api/browse?table=herbs&1%3D1%20UNION%20SELECT...=x`
- **修复**：列名白名单（`PRAGMA table_info` 取合法列）

### 🔴 P0-SEC-3　server.py SQL 注入 —— `api_detail`/`api_filter_options`/`api_export_table`/`api_export_search`
- **证据**：`:132` `f"SELECT * FROM {table} WHERE id = ?"`、`:296` `f"SELECT DISTINCT {column} FROM {table}..."`、`:303` `f"SELECT * FROM {table}"`、`:348` `FROM {table}`
- **状态**：代码层结构漏洞【已确认】；URL 路径利用（需绕过 %20 编码）【需运行时验证】
- **修复**：统一白名单覆盖所有 table/column 拼接点

### 🔴 P0-SEC-4　server.py 目录穿越 —— `/screenshots/`
- **证据**：`server.py:833-835` `rel = path[len('/screenshots/'):]` → `file_path = SCREENSHOTS_DIR / rel` → `if file_path.exists() and file_path.is_file()` —— **无 resolve()/startswith 校验**
- **触发**：`GET /screenshots/../../../etc/passwd`
- **影响**：读取服务器任意可读文件
- **修复**：`(SCREENSHOTS_DIR/rel).resolve()` 后校验 `startswith(SCREENSHOTS_DIR.resolve())`

### 🔴 P0-ETL-1　`populate.py` `build()` 是空壳 —— README 重建命令失效
- **证据**：`populate.py:90-107`，`build()` 内只有 print，`:102` 注释 `# ... (完整构建逻辑在 GitHub 仓库中)`，`:104` `print("⚠️ 请使用 --download 先下载源仓库...")`
- **影响**：README 宣称的 `python3 populate.py --rebuild` **不能重建数据**；`--download`（download_repos, :31-42）完整可用。当前库的数据实际由 `import_new_cases.py` + `enrich_*.py` + `populate_*.py` 填充，**无统一重建入口**。
- **定位**：formulas 重复的来源脚本尚未定位（build() 空壳，重复应由建库时的导入脚本造成，Phase 0 需用 git log 定位）

## 5. RAG 推理链断点

```
症状 ──89%──✅──> 证型 ──6.2%──🔴断──> 方剂 ──26%──❌──> 药物
                                                   ↘ 17% ❌ 医案证据
```

- **症状→证型**：通畅（syndrome_symptoms 89%）。推理起点可靠。
- **证型→方剂**：**致命断裂**（19/305）。286 个方剂无证型关联，无法从证型检索方剂。归因：倪师原文系统性「证-方」关联未提取，或 ETL 未抽取。
- **方剂→药物**：严重不足（26%），与 composition 73% 空一致。
- **方剂→医案**：严重不足（17%），叠加 herbal_rx 54% 空，医案作证据效力弱。
- **叠加因素**：formulas 重复（同方多 id）+ 六经 81% 空 → 即便补方-证，方剂侧标识与维度也紊乱。
- **结论**：**当前数据无法完成一条普遍可用的「症状→证型→方剂」可解释推理**，仅对极少数（19 个方剂涉及的）证型能走通。

## 8. 最小实施路线（Phase 0–3）

### Phase 0 — 可信基线（只读产出，不改数据）
- `scripts/audit_db.py`：一条命令输出表行数/孤儿/重复关系/JSON 错误/空字段率/覆盖率
- `tests/test_api.py`：标准库 unittest，覆盖成功/空结果/非法参数/边界分页
- README 数字改为"由 audit_db.py 生成"或标注统计日期
- 定位 formulas 重复插入来源（git log + 建库脚本）

### Phase 1 — 工程化加固（不改 DB）
- `config.py`：统一 DB_PATH/SOURCE_ROOT，支持环境变量 `TCM_DB_PATH`/`TCM_SOURCE_ROOT`
- `etl.py`：协调入口（dry-run + logging + 顺序 + 每步统计）
- 各脚本接入 config；import_new_cases 改幂等

### Phase 1.5 — 安全 P0（与 Phase 1 并行，不改 DB）
- server.py 白名单（table/column）+ screenshots resolve 校验 + 其他 P1

### Phase 2 — 推理骨架补全（**需确认**，批量写库）
- 顺序：① formulas 去重合并（先迁关系）② 六经补全（合并 nihaixia 版 + 原文）③ 方-证关系（180-220 方，350-600 条带 evidence）④ 证-症质量复核 ⑤ 方-药 ⑥ 经络-穴位
- 新增 `evidence` 表 + `entity_aliases` 表（Schema 迁移）
- 方-证每条含 source_type/source_id/evidence_text/confidence/extraction_method/reviewed

### Phase 3 — 结构化查询（不改 DB，加端点）
- `/api/reason`、`/api/formulas/by-syndrome`、`/api/syndromes/by-symptom`、`/api/evidence`
- 返回候选证型/方剂/匹配路径/证据/来源/置信度/缺失信息/高风险提示
- 检索：LIKE + aliases 同义词 + 多字段加权 + 结构化 JOIN（**不上向量库**）；FTS5 仅作可选增强且须解决中文分词

## 七个最终问题回答

**1. 经过复现的 P0/P1 bug？**
P0：SQL 注入 4 处（server.py:120/123/124/132/296/303/348）、目录穿越（:832-840）、populate.py build() 空壳（:90-107）。P1：LIKE 未转义、分页无边界、异常泄露、连接未 finally、外键运行时未启用、clinical_cases 裸 INSERT。**均经主线程代码/SQL 复核**。

**2. 四方向优先级是否调整？**
同意用户框架（执行序 A>B>C>D；RAG 贡献 C>D>B>UI），**但补充：server.py 安全 P0 必须插到最前**——本地服务若绑 0.0.0.0，注入+CORS 可被读取全库。安全修复不改 DB、低风险，应优先于数据补全。实际顺序：**安全 P0 → Phase 0 基线 → Phase 1 工程化 → Phase 2 数据 → Phase 3 查询**。

**3. 数据补全的前置条件？**
① 外键启用 + clinical_cases 幂等（防补数据时引入孤儿/重复）② config 化（可回滚环境）③ evidence/aliases 表（补的关系要有证据落点）④ **formulas 去重**（否则方-证会挂到重复 id 上，加剧混乱）。

**4. 方-证关系优先覆盖哪批方剂？依据？**
**有医案使用记录的方剂优先**（case_formulas 294 条涉及的方剂）+ 高频经方（桂枝汤/麻黄汤/小柴胡汤/白虎汤/真武汤等）。依据：这些方剂有临床证据可填 evidence_text，且 `formulas.is_high_risk` 需重点覆盖。目标 180-220 方剂（60-70%），不追求 305/305——无可靠证据的保持"未知"。

**5. 现有数据能否完成一条可解释的症状→证型→方剂？**
**部分能，但不普遍**。症状→证型通（89%），但证型→方剂仅 19 条，只能对极少数证型走通。需补方-证后才普遍可用。示范链可跑，但覆盖窄。

**6. 下阶段 2-4 周最小交付？**
Phase 0（audit_db.py + test_api.py + README 修正）+ Phase 1（config.py + etl.py + clinical_cases 幂等）+ Phase 1.5（server.py 安全 P0）+ Phase 2 起步（formulas 去重 + 六经补全 + 方-证首批 ~100 条带 evidence + evidence/aliases 表）。即"可信基线 + 工程化 + 安全 + 推理骨架起步"。

**7. 哪些建议属过度工程化应放弃？**
向量库/FAISS、微服务/JWT/K8s、ORM、Web 框架、任务队列、"智能诊断算法"（第一版只做可解释候选检索）、为重构而重构、无原文依据的模型补全、给单页 H5 套企业架构。（均与项目「轻量 + 知识编译器」定位冲突。）
